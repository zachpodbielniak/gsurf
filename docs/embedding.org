#+TITLE: Embedding gsurf
#+OPTIONS: toc:2 num:nil

* libgsurf

gsurf is library-first: link =libgsurf.so= and use its GObject API. A
=gsurf.pc= pkg-config file is installed.

#+begin_src sh
pkg-config --cflags --libs gsurf
#+end_src

* Minimal embed

#+begin_src c
#include <gsurf/gsurf.h>

int main(int argc, char **argv)
{
    gsurf_init(&argc, &argv);
    gsurf_backend_init(&argc, &argv, NULL);

    GsurfConfig *config = gsurf_config_new();
    GsurfApplication *app = gsurf_application_new(config);

    GsurfWindow *window = gsurf_window_new();   /* compiled-in backend */
    GsurfView   *view   = gsurf_view_new();
    gsurf_window_add_view(window, view);
    gsurf_application_add_window(app, window);

    gsurf_view_load_uri(view, "https://example.com");
    gsurf_window_present(window);

    gsurf_application_run(app);   /* or drive your own GLib main loop */
    return 0;
}
#+end_src

* Loading modules (host-controlled)

The library never loads modules on its own. A host decides *where* modules come
from by configuring the module manager's search path and then asking it to load:

#+begin_src c
GsurfModuleManager *mgr = gsurf_module_manager_get_default();
gsurf_module_manager_set_application(mgr, app);
gsurf_module_manager_set_config(mgr, config);

/* Point at the embedding application's OWN module directory only. */
gsurf_module_manager_add_search_path(mgr, "/opt/cmacs/lib/gsurf/modules");
gsurf_module_manager_load_modules(mgr);          /* loads *.so from each path */

gsurf_module_manager_activate_all(mgr);          /* after window/view exist */
#+end_src

This is the isolation an embedder (e.g. cmacs) wants: an embedded gsurf loads
=*only*= the modules the host points it at. The =GSURF_MODULE_PATH= environment
variable and the built-in system module directory are =never= consulted
automatically — that env/exe/system fallback is the standalone =gsurf= binary's
own policy (in its =main.c=), not the library's. So an embedded gsurf will not
pick up a system-installed gsurf's modules, even if one is installed.

Useful calls (see =gsurf-module-manager.h=):

- =gsurf_module_manager_add_search_path()= — append a directory (deduped, ordered).
- =gsurf_module_manager_set_search_paths()= — replace the whole list (or clear with %NULL).
- =gsurf_module_manager_clear_search_paths()= — empty the list.
- =gsurf_module_manager_get_search_paths()= — inspect the current list.
- =gsurf_module_manager_load_modules()= — load =*.so= from every path, first-match
  by file name wins (like =$PATH=).
- =gsurf_module_manager_get_system_module_dir()= — the compile-time system dir,
  returned for hosts that want to opt in to system modules by adding it explicitly.

* Embedding the widget in another GTK app

=gsurf_view_get_native_widget()= / =gsurf_window_get_native_widget()= return the
underlying =GtkWidget*= as an opaque =gpointer= so the abstract API never leaks a
GTK type. A GTK-aware host casts it and packs it into its own container instead
of calling =gsurf_window_present()=. With the webkit2gtk-4.1 backend the
returned widget *is* the =WebKitWebView=, so a host can reach the WebKit layer
directly (e.g. =webkit_web_view_get_context()=, =..._get_user_content_manager()=)
to add features that the abstract API does not wrap.

* View signals a host can observe

A =GsurfView= emits these GObject signals; a host that parents the raw view
(no =GsurfWindow=) connects to them for chrome/automation:

- =load-changed= =(GsurfLoadEvent)=, =uri-changed= =(uri)=, =title-changed= =(title)=
- =progress-changed= =(gdouble 0..1)=, =hovered-uri-changed= =(uri or "")=
- =favicon-changed= =()=, =web-process-terminated= =()=
- =create-view= =(uri) -> GsurfView*= — a popup / =target=_blank= / =window.open=
  request. The handler returns a *new, unrealized* =GsurfView= for WebKit to
  take over, or =NULL= to block the popup. The return is =(transfer none)=, and
  it uses a first-handler-wins accumulator. A host that has no spare unrealized
  view should return =NULL= and open the URI itself instead — returning a view
  that is already realized/parented fights WebKit's popup lifecycle.

* Downloads and permissions (host dispatch)

These are answered *synchronously*, so a host hook must decide without blocking:

- *Downloads*: the backend connects =download-started= on the WebKit context and
  dispatches =decide-destination= to a =GsurfDownloadHandler= module
  (=(uri, suggested) -> path=, first non-NULL wins; return without a path to let
  the engine default). gsurf tracks no further progress itself — a host that
  wants progress/finished/failed events listens on the WebKit download directly
  (via the default =WebKitWebContext=).
- *Permissions*: the backend dispatches =permission-request= to a
  =GsurfPermissionHandler= module expecting a synchronous verdict
  (=ALLOW=/=DENY=/=PROMPT=); =PROMPT= returns =FALSE= so the engine default
  applies, which also lets a host that connects its *own* =permission-request=
  handler on the =WebKitWebView= (and returns =TRUE=) win when no gsurf
  permission module is enabled. A blocking modal inside this callback is unsafe;
  decide from a policy, or defer by reffing the request and answering later.

* GObject Introspection

=make BUILD_GIR=1= produces =Gsurf-0.1.gir= / =Gsurf-0.1.typelib=, so gsurf can be
called from any introspected language. The abstract API exposes no GTK/WebKit
types, keeping the typelib backend-agnostic.

* cmacs (future)

gsurf's clean GLib/GObject API and GTK-free abstract surface are designed so a
=GsurfView= widget can later be embedded as a buffer/overlay in the cmacs Emacs
fork via GObject Introspection. The library makes no GTK-main-loop assumption
(it uses the default GLib main context), which a host event loop can integrate.
